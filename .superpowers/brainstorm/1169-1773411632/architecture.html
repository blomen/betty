Bankroll Optimizer — System Architecture

Rolling horizon planning with Monte Carlo simulation

Full System Flow
CURRENT STATE (Real Data)
Provider balances & wagering progress
M2 limitation risk per provider
Available edges from latest scan
Bonus configs (providers.yaml)
Historical edge distributions (DB)
SIMULATOR (Forward Planning Engine)
Takes snapshot of current state
Simulates N trajectories (1000+) forward
Each trajectory: random edge arrivals + limitation events
Models: deposits, bet routing, wagering progress, limitations
Terminal metric: bankroll growth at horizon (30/60/90 days)
MONTE CARLO PLANNER (Decision Engine)
Evaluates action candidates:
  → "Deposit 1000 at Betinia next"
  → "Route this bet to Unibet (wagering)"
  → "Withdraw from ComeOn, redeploy to 10bet"
Each action → simulate 1000 trajectories → avg terminal bankroll
Pick action with highest expected growth
RECOMMENDATIONS (Output)
● Next deposit: Provider X, amount Y
● Bet routing priority: [Provider A > B > C] for same-edge bets
● Fund transfers: Move Z from cleared provider to active
● Freebet target: Best edge at freebet provider
● Alerts: "Provider Q at 80% limitation risk, 40% wagering left"
RE-PLANNING TRIGGERS
New extraction scan completed
Bonus status changed (cleared/expired)
Provider limited (detected via risk score)
Manual deposit/withdrawal
Periodic (every 6 hours)

Key Design Decisions

Why Monte Carlo Planning (not pure RL)

  • Plans from ACTUAL current state — no generalization needed
  • Incorporates latest M2 predictions in real-time
  • No training phase — works immediately with simulator
  • Transparent: "I simulated 1000 futures, this path wins"
  • Handles non-stationary environment (new providers, changing edges)

Trade-offs

  • Compute cost: 1000+ trajectories per decision point
  • Simulator fidelity: only as good as the models feeding it
  • No learned intuition — pure brute force search
  • Action space must be discretized (deposit amounts, routing)
